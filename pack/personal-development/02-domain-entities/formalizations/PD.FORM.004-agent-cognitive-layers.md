# 5-сущностная онтология агента (Agent Cognitive Layers)

## 1. Концепция

Минимальная онтология, разрешающая путаницу между артефактами, способностями, процедурами, рамками интерпретации и процессами изменения агента.

**Ключевое утверждение**: один и тот же «исходник» (описание) собирается в разные «программы» (навыки) у разных агентов, потому что различаются «компилятор» (мировоззрение) и «среда исполнения» (контекст).

## 2. Пять сущностей

| # | Сущность | Тип | Где живёт | Программная аналогия |
|---|----------|-----|-----------|---------------------|
| 1 | **Описание** (Description) | Информационный артефакт | В репозиториях, текстах | Исходники / документация |
| 2 | **Знание** (Knowledge) | Диспозиция агента (capability) | В агенте | — |
| 3 | **Навык** (Skill) | Исполняемая процедура агента | В агенте | Исполняемая программа |
| 4 | **Мировоззрение** (Worldview) | Мета-рамка интерпретации | В агенте | Семантика + система типов + правила сборки |
| 5 | **Обучение** (Learning) | Процесс изменения агента | Происходит с агентом | Компиляция / сборка |

Плюс **контекст исполнения** (тело, среда, ресурсы, социальные ограничения) = «железо / ОС».

## 3. Определения (FPF A.7 Strict Distinction)

### 3.1. Описание (Description)

Артефакт в знаковой форме: текст, формула, схема, код, промпт.

- **Свойства**: копируемость, переносимость, читаемость
- **Примеры**: FPF, SPF, Pack, методичка, код-репозиторий
- **Важно**: описание само по себе ничего не делает — оно используется агентом

### 3.2. Знание (Knowledge as capability)

Способность агента устойчиво достигать результата при заданных условиях, с определённым качеством.

- **Тип**: диспозиционное свойство агента
- **Проверяется**: по выполнению задач, не по качеству формулировок
- **Пример**: «умеет проводить переговоры так, что конверсия ≥ 20% при условиях A, B, C»
- **Знание шире навыка**: включает модели, критерии, стратегии

### 3.3. Навык (Skill)

Исполняемая процедура в агенте: автоматизированная последовательность операций/решений, запускающаяся в ситуации.

- **Тип**: внутренняя процедура агента (executable)
- **Отличие от знания**: навык — процедурный слой, знание включает стратегический
- **Пример**: «проводит интервью по структуре, задаёт уточняющие вопросы, фиксирует сигналы»

### 3.4. Мировоззрение (Worldview)

Рамка интерпретации и отбора, определяющая:

- что считать «реальным» и «причинным» (онтологические обязательства)
- что считать «доказательством» (эпистемические нормы)
- что считать «ценным» (целевые функции/ценности)
- какие объяснения и методы допустимы

**Мировоззрение — не описание**. Это «компилятор», определяющий, как агент превращает описания в действия и какие действия считает правильными.

### 3.5. Обучение (Learning)

Процесс изменения агента, в котором описания, практика и обратная связь трансформируются в способности/навыки.

- **Тип**: процесс (не система, не курс, не артефакт)
- **Результат**: изменение capabilities/skills и частично мировоззренческих настроек
- **Важно**: обучение как процесс ≠ обучение как курс/программа (см. bounded context)

## 4. Три точки потерь (Failure Modes перехода)

| Переход | Что теряется | Причина | Компенсация |
|---------|-------------|---------|-------------|
| Описание → Интерпретация | Контекст, условия применимости | Разные мировоззрения читают по-разному | Distinctions, failure modes, glossary |
| Интерпретация → Навык | Процедурность | Без практики «сборка» не происходит | Practice, feedback loops |
| Навык → Исполнение | Предсказуемость | Среда меняет результат | Testing, adaptation |

## 5. Маппинг на FPF/SPF/Pack

| Уровень архитектуры | Тип сущности | Роль |
|---------------------|-------------|------|
| FPF | Описание (мета-принципов) | Наиболее абстрактные правила формализации |
| SPF | Описание (формы и процесса) | Правила создания Pack'ов |
| Pack | Описание (доменного знания) | Source-of-truth предметной области |
| Downstream | Описание (реализации) + контекст исполнения | Код, курсы, агенты |
| **Агент** | **Знание + Навыки + Мировоззрение** | Где описания становятся способностями |

**Вывод**: FPF, SPF, Pack — все являются описаниями. Они содержат знание в артефактной форме. Но знание как capability живёт в агенте.

## 6. Проверяемое утверждение

> Один и тот же «исходник» (описание) может «собраться» в разные «исполняемые программы» (навыки) у разных агентов, потому что различаются «семантика/валидация» (мировоззрение) и «среда исполнения» (контекст).

Это снимает парадокс «если знание невыразимо, зачем писать тексты»: тексты — входные артефакты, а знание/навык — результат корректной «сборки» через обучение.

## 7. Инженерный подход: контракты вместо метафизики

Для инженерного обсуждения (включая ИИ) фиксируй контракт:

1. **Что является описанием?** → формат артефакта (Pack, код, промпт)
2. **Какова семантика?** → какие различения приняты (FPF, SPF)
3. **Какие тесты валидируют знание?** → задачи, метрики, условия
4. **Какой процесс обучения?** → как описание становится способностью

Это превращает спор из «выразимо/невыразимо» в проверяемые вопросы.
